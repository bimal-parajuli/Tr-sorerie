Prisma ORM — Schema, Migrations, Transactions
Critical
▼
Prisma is your daily driver for all DB interaction. Know it well — especially its transaction API, which is your primary guarantee of atomicity.

tRPC — Router Design and Middleware
Important
▼
tRPC isn't complex — but understanding its middleware chain and context pattern is essential for cleanly enforcing access control across 50+ procedures.
Context creation (createTRPCContext)
Middleware chain — protectedProcedure, adminProcedure
Router composition — merging sub-routers
Error handling — TRPCError codes
Input/output type inference
Batching and request deduplication
Pattern to design early: a middleware hierarchy where publicProcedure → authedProcedure → employeeProcedure → adminProcedure each add role checks. Every router procedure picks the right level. Changing a role rule means changing one middleware, not hunting 50 procedures.
Supabase — Row Level Security (RLS)
Important
▼
RLS is a database-layer access control policy. When enabled, a member querying savings_accounts literally cannot see another member's row — the DB filters it out. It's a second line of defence behind your application layer.
ENABLE ROW LEVEL SECURITY on tables
CREATE POLICY — FOR SELECT, INSERT, UPDATE
auth.uid() and auth.jwt() in policy conditions
Service role key bypasses RLS — understand when to use it
Policy performance implications
Decision to make: whether to enforce all access control in tRPC middleware (simpler) or add RLS as a defence-in-depth layer too (more secure, more setup). For a financial app, do both.
Start → Supabase docs: "Row Level Security" — has good worked examples
Financial Data Patterns
Critical
▼
Financial software has specific, non-obvious patterns that general web dev tutorials never cover. Getting these wrong means undetectable data corruption.
Never use floats for money — always NUMERIC/Decimal
Immutable ledger pattern (append-only transactions)
Reversals, not deletions or edits
Idempotency keys — prevent duplicate transactions on retry
Reducing balance EMI formula
Optimistic locking vs pessimistic locking
Reconciliation patterns
Audit trail design
The idempotency key pattern is especially important: if a deposit API call times out and the client retries, you must not create two deposit records. The key (a UUID generated by the client) guarantees the server is idempotent.
Start → Stripe's engineering blog on idempotency — the best public resource on this topic

Architecture Patterns
High Leverage
Repository Pattern — Separating DB Logic from Business Logic
Important
▼
Without this pattern, your tRPC handlers will turn into 100-line blobs of Prisma calls mixed with business rules. As the app grows, you'll be unable to test the logic without touching the DB.
Thin tRPC handlers that call service functions
Service layer holds business rules
Repository layer holds DB queries
Dependency injection for testability
When this pattern is overkill (early stages)
Suggested structure: router → service → repository. The tRPC router handles HTTP/auth. The service (e.g. depositService.deposit()) holds business rules. The repository does the Prisma query. Each layer is independently testable.
Error Handling Architecture — Typed Errors
Important
▼
Generic "something went wrong" errors in a financial app mean employees can't diagnose what failed. An insufficient balance error and a network error need different UI responses.
Custom error classes (InsufficientBalanceError, LoanAlreadyClosedError)
TRPCError with code and message
Discriminated union Result types (success | error)
Prisma error codes (P2002 unique, P2025 not found)
neverthrow library — railway-oriented error handling

Feature Flags & Incremental Rollout
Useful
▼
Your Phase 2 and Phase 3 features need to be built, tested, and merged into the main codebase without being visible to users. Feature flags enable this without long-lived branches.
Environment variable based flags (simplest)
Role-based feature access (employees see Phase 2 features first)

Auth & Security
Non-Negotiable
JWTs — What They Are and Where They Can Go Wrong
Critical
▼
Supabase Auth issues JWTs. Your entire access control system runs on verifying these tokens correctly. Misverification = unauthorized access to financial data.
JWT structure — header, payload, signature
Signature verification — never skip it
Token expiry and refresh token rotation
Where to store tokens (httpOnly cookies vs localStorage)
Role claims in JWT payload
Algorithm confusion attacks (alg: none)
Rule: never store JWTs in localStorage. Always use httpOnly cookies. Supabase's SSR helper handles this for Next.js — understand why it does it this way.
Start → jwt.io has a good explainer. Then read: "Stop using JWT for sessions" (article) to understand tradeoffs.
RBAC — Role-Based Access Control Implementation
Critical
▼
Your app has 3 roles with a strict permission matrix. Understand the right places to enforce this — and why enforcing it in only one place is insufficient.
Middleware-level route protection (Next.js middleware)
tRPC procedure-level enforcement
DB-level via Supabase RLS
Defence in depth — all three layers together
ABAC (attribute-based) vs RBAC — when RBAC isn't enough
Concrete risk: if you only protect routes in the UI and a tRPC procedure is left unprotected, any authenticated user can call it directly via a fetch() call. The API layer must always enforce roles independently of the UI.
OWASP Top 10 — Relevant Subset for Financial Apps
Important
▼
You don't need to study all 10 equally. These are the ones that directly apply to your app's architecture.
Broken Access Control — #1, directly relevant
Injection (SQL injection via raw queries)
Insecure Direct Object References (IDOR) — fetching another member's data by ID

IDOR is particularly relevant: if your GET /account/:id endpoint doesn't check that the requesting user owns that account, any member can view any account by guessing UUIDs. This must be checked at the tRPC layer, not just the UI.

Ops & Deployment
Production Readiness
Database Migrations in Production
Critical
▼
Running prisma migrate deploy on a live financial DB without a plan is how you cause downtime or data loss. This is different from running migrations in development.
prisma migrate dev vs migrate deploy
Expand/contract pattern for zero-downtime migrations
Never drop columns or rename them in one step
Locking behaviour of ALTER TABLE in Postgres
Migration rollback strategies
Baseline migrations for existing databases
The expand/contract pattern: to rename a column, first add the new column (expand), backfill the data, update the app to use the new column, then drop the old one (contract). Doing it in one migration locks the table and drops data if anything fails.
