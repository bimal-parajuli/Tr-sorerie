Architecture Overview

How the pieces connect, and what calls what.

📱
Phase 3 native apps call the same tRPC API and use the same Supabase Auth SDK. No new backend needed — just a new client.

Frontend Framework

The biggest decision. Determines your Phase 3 native path.

⚡
Next.js 15 (App Router) Recommended
React framework — SSR + RSC + PWA + API routes

For

  • React base = React Native in Phase 3 with shared code
  • App Router supports server components for fast dashboard loads
  • API Route Handlers serve as your backend in Phase 1
  • Vercel deploys it in one click, scales globally
  • Best-in-class TypeScript support
  • Huge ecosystem — every library has a Next.js guide

Against

  • App Router has a learning curve if you're used to Pages Router
  • Server Components add mental overhead early on
  • Cold starts on Vercel free tier (negligible for internal app)
✓ Chosen. React foundation is non-negotiable for Phase 3 native continuity. Next.js specifically adds SSR, built-in routing, and API layer — removing the need for a separate backend project in Phase 1.
🏃
Vite + React (SPA) Alternative
Pure client-side, no SSR

For

  • Simpler mental model than App Router
  • Fastest local dev experience
  • React Native still possible in Phase 3

Against

  • No built-in API routes — need a separate backend from day one
  • Slower initial load (no SSR) — bad for mobile in Nepal
  • More repo complexity
Skip. Next.js gives everything Vite gives + SSR + API layer for free.
🚫
Vue / Nuxt or Svelte / SvelteKit Avoid
Different rendering paradigm

For

  • Both are excellent frameworks
  • SvelteKit is notably fast

Against

  • Phase 3 native means React Native or Flutter — neither uses Vue/Svelte
  • You'd rewrite your entire UI layer for native apps
  • Ecosystem is smaller for financial app tooling
Hard no. Creates a dead end at Phase 3. Good frameworks, wrong context.

Backend Layer

Where your business logic and DB transactions live.

🔗
Next.js Route Handlers + tRPC Recommended
Monorepo — frontend and API in one Next.js project

For

  • One project, one deploy, one language
  • tRPC gives 100% type-safe API — no manual type syncing between front and back
  • Shared Zod schemas for validation on both sides
  • When Phase 3 needs a standalone API, you extract the tRPC router to a separate service — no logic rewrite
  • No CORS configuration, no API versioning overhead in Phase 1

Against

  • Next.js Route Handlers are stateless — no persistent worker threads
  • Background jobs (e.g. nightly interest cron) need an external scheduler
  • Vercel function timeout limit (10–60s depending on plan)
✓ Chosen. The type safety from tRPC alone saves days of bug hunting in a financial app. Background jobs are handled by Supabase pg_cron or a tiny separate cron service.
🛠️
Separate Node.js + Fastify / Express Alternative
Dedicated API server, separate repo/deploy

For

  • More control over server lifecycle, connection pooling
  • Better for long-running processes and WebSockets
  • Familiar REST pattern if team knows it

Against

  • Two projects to maintain, two deploys, two CI pipelines
  • Manual type sharing between frontend and backend
  • No productivity gain in Phase 1 for the extra complexity
Worth revisiting in Phase 3 if you need persistent connections or high-volume background processing. Not now.

Database Hosting

Your schema is PostgreSQL. The question is where to host it.

🟢
Supabase Recommended
Managed PostgreSQL + Auth + Storage + Realtime in one project

For

  • Auth, Storage, and DB are one integrated project — single dashboard, single billing
  • Row Level Security (RLS) lets you enforce access control at the DB layer — your role model maps directly
  • Supabase Auth gives you phone OTP + Google + Apple with native SDK that works on React Native too
  • Free tier: 500MB DB, 1GB storage, 50,000 MAU auth — plenty for Phase 1
  • Singapore region available (low latency from Nepal)
  • pg_cron extension available for scheduled interest jobs

Against

  • Pausing on free tier after 1 week of inactivity (non-issue on paid)
  • Prisma + Supabase requires using the direct connection string, not the pooler, for migrations
✓ Chosen. Supabase consolidates 3 services (Auth, DB, Storage) that you'd otherwise wire together separately. The RLS feature is genuinely powerful for a financial app with strict access tiers.
🚂
Railway Alternative
Managed Postgres, simpler than Supabase

For

  • Simpler — just a Postgres instance, no opinionated SDK
  • Persistent connections, no pausing
  • Good for if you want full control

Against

  • No built-in Auth, Storage, or Realtime — you'd integrate those separately
  • More setup work
Good option if you want to use Firebase Auth separately and just need a Postgres host. More work to set up than Supabase all-in-one.
🚫
Firebase Firestore as primary DB Avoid
NoSQL document database

For

  • Great phone OTP auth
  • Realtime out of the box

Against

  • No ACID transactions spanning multiple collections — fatal for a financial ledger
  • Your relational schema doesn't map to documents cleanly
  • Reporting queries (SUM, GROUP BY) are painful or impossible
Hard no for the database. Firebase Auth is fine, but Firestore cannot replace PostgreSQL for financial data. The atomic transaction requirement alone rules it out.

API Layer Design

How the frontend talks to the backend, and why type safety matters in a financial app.

🔒
tRPC Recommended
End-to-end type safety — no REST, no GraphQL

Why tRPC for a financial app

  • A deposit endpoint that accepts a wrong type fails at compile time, not at runtime
  • Shared Zod input schemas — same validation on frontend form AND backend handler
  • Auto-generated TypeScript types from your router — no OpenAPI, no codegen steps
  • Procedure-level auth middleware — attach role check to every sensitive route declaratively
  • Works with TanStack Query natively — mutations and queries just work

Tradeoff

  • Not a public REST API — Phase 3 native apps must use the tRPC client too (they will, via the shared API package)
  • Third-party integrations (e.g. payment gateway webhooks) need plain Route Handlers alongside tRPC — fine
✓ The type safety dividend compounds. Every time you change a DB field, TypeScript immediately shows you every callsite that breaks. In a financial app, this is worth a lot.
// server/routers/savings.ts — example tRPC router
export const savingsRouter = router({
  deposit: protectedProcedure
    .input(z.object({
      accountId: z.string().uuid(),
      amount:    z.number().positive(),
      note:      z.string().optional(),
    }))
    .mutation(async ({ ctx, input }) => {
      // ctx.user.role is typed — guaranteed EMPLOYEE or ADMIN
      return await prisma.$transaction(async (tx) => {
        const account = await tx.savingsAccount.findUniqueOrThrow(...)
        await tx.savingsTransaction.create({ ... })
        await tx.savingsAccount.update({ balance: account.balance + input.amount })
      })
    }),
})
// client — fully typed, no manual API calls
const deposit = trpc.savings.deposit.useMutation()
deposit.mutate({ accountId: '...', amount: 5000 })
// TypeScript knows the exact input shape and return type

State Management

Two different concerns — server data and UI state — handled by two small libraries.

🔄
TanStack Query Server State
Async data fetching, caching, refetch

Why

  • Handles loading / error / stale states automatically
  • Optimistic updates for deposit/payment flows
  • Cache invalidation — after a deposit, auto-refetch the account balance
  • Carries over to React Native with the same API
  • tRPC uses it internally — you get it for free
🐻
Zustand UI State
Modal state, sidebar, session context

Why

  • 200-line library — no boilerplate, no providers, no reducers
  • Used for: open drawer state, active member context, confirmation modals
  • Works the same in React Native
  • Don't use it for server data — that's TanStack Query's job
🚫
Don't use Redux. TanStack Query replaces most of what Redux was used for (server data). Zustand handles the rest. Redux is overkill here and adds 3x the boilerplate for no benefit.

Phase 3 — Native Apps

The strategy that makes this stack decision matter long-term.

📱
Expo (React Native) Recommended
One codebase → Android + iOS

For

  • React — same component model as your Next.js app
  • tRPC client package is shared verbatim with the web app
  • TanStack Query, Zustand, Zod — all work unchanged
  • Supabase Auth SDK has an Expo-specific package
  • Expo EAS Build publishes to Play Store and App Store
  • You can share a /packages/api folder between Next.js and Expo in a monorepo

Against

  • UI components don't transfer — you rewrite screens in React Native primitives (but logic transfers)
  • Native modules (biometrics, camera) need Expo SDK
✓ 60–70% of your business logic (API calls, validation, state management) transfers without changes. You rewrite only the UI layer.
🏗️
Monorepo structure for Phase 3 — use Turborepo with apps/web (Next.js), apps/mobile (Expo), and packages/api (tRPC client + Zod schemas). Set this up from day one — it costs nothing and makes Phase 3 trivial.
# Turborepo monorepo layout
apps/
  web/          # Next.js PWA
  mobile/       # Expo app (Phase 3)
packages/
  api/          # tRPC router types + Zod schemas (shared)
  db/           # Prisma schema + client (server-only)
  ui/           # Shared non-visual logic, utils

Deployment

Simple, proven, cheap at SACCO scale.

▲
Vercel Frontend + API

Why

  • Next.js is made by Vercel — zero config deploy
  • Edge network includes Singapore (ap-southeast-1)
  • Preview deployments per branch — test before merging
  • Free tier handles Phase 1 easily. Pro is $20/mo.
  • Environment variables, cron triggers, log drains — all built in
🟢
Supabase DB + Auth + Storage

Why

  • Already decided above. Handles DB, Auth, Storage.
  • Free: 500MB DB. Paid: $25/mo for 8GB + no pausing
  • Daily backups on paid. Point-in-time recovery on Pro.
  • Singapore region
💡
When to upgrade from Supabase free tier: Once you go live with real member data, move to Supabase Pro ($25/mo). The free tier pauses after 1 week of inactivity — fine for development, not for production.
