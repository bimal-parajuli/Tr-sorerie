Data Model Design — Nepal SACCO

Schema & Architecture
Blueprint

A resilient, extensible database design for core SACCO operations — savings, lending, and expense management — built for progressive feature rollout across PWA and native clients.

📦 13 Core Tables 🔐 3 Access Tiers 🧩 PostgreSQL / Firebase Firestore Compatible 📱 PWA → Android → iOS

Design Principles

The schema is built around five non-negotiable properties for a financial system operating in a trust-sensitive cooperative environment.

🔁
Immutable Ledger
Transactions are never deleted, only reversed

Every financial movement (deposit, payment, interest) creates a permanent record with a balance_before and balance_after snapshot. Corrections are done via reversal transactions, never edits.

⚡
Strategic Denormalization
Running balances cached for instant reads

Account balances, outstanding loan principal, and total paid amounts are stored as computed-and-cached fields, updated atomically via transactions. No expensive SUM() queries on hot paths.

🧱
Atomic Operations
DB transactions wrap every mutation

A deposit must simultaneously create a savings_transaction row AND update savings_accounts.balance. Both succeed or both fail — no partial states.

🏗️
Soft Deletes
Status fields replace hard deletion

Users, loans, accounts are never hard-deleted. Status enums (ACTIVE / CLOSED / SUSPENDED) control visibility. Historical data is always preserved for audits and reporting.

📋
Full Audit Trail
Every mutation logged with actor & diff

A write-only audit_logs table captures who did what, when, and the exact before/after JSON diff. Critical for cooperative governance and dispute resolution.

🔌
Extensible Metadata
JSON columns absorb future flexibility

Key tables carry a metadata JSONB column — a flexible escape hatch for future features (tags, custom fields, channel-specific data) without schema migrations.

Auth & Identity

All users share a single users table with a role discriminator. Authentication is decoupled from profile data — supporting multiple auth providers per user, with phone as the mandatory identifier.

💡
Phone-first identity Phone number is the canonical unique identifier. Google/Apple OAuth tokens are linked to it but are supplementary. This ensures a member can always recover access via OTP even if they lose access to their OAuth provider.

Savings Accounts & Transactions

The double-entry savings ledger. Every monetary change is an immutable transaction row; the account balance is a cached aggregate that must stay in sync via atomic DB transactions.

⚠️
Critical: Atomic balance updates Application logic MUST wrap every deposit/withdrawal in a database transaction that (1) inserts a savings_transactions row and (2) updates savings_accounts.balance. Never update the balance without creating a corresponding transaction record.

Loan Lifecycle

Loans move through a structured pipeline from application → approval → disbursement → repayment. Four tables capture the full lifecycle with pre-calculated EMI schedules for deterministic reporting.

📐
Pre-computed EMI Schedule When a loan is disbursed, the application generates the full loan_emi_schedule (all N installments) using the reducing-balance formula. This enables instant "how much is due this month" and "show me all overdue EMIs" queries without any runtime calculation.

Denormalization Strategy

Computed fields are stored alongside their source data to eliminate expensive aggregates on hot read paths. Consistency is maintained via atomic writes — not eventual reconciliation.

Cached Fields & Update Triggers
TableCached ColumnUpdated WhenGuard
savings_accountsbalanceOn every savings_transaction INSERTSame DB transaction. Rollback if either fails.
savings_accountslast_transaction_atOn every savings_transaction INSERT
loansoutstanding_principalOn every loan_payment INSERT= principal_amount − total_paid_principal
loanstotal_paid_principal / interest / penaltyOn every loan_payment INSERTAccumulated totals.
loanslast_payment_dateOn every loan_payment INSERT
loan_emi_schedulepaid_amount, paid_at, statusWhen a loan_payment allocates to this EMI rowCovered by same transaction as loan_payments INSERT.
savings_transactionsbalance_before / balance_afterAt INSERT time, read current balance, computeSnapshot approach — never re-derived later.
🔄
Reconciliation Job (optional, nightly) A background cron can verify that cached balances match the SUM of their transaction histories. Discrepancies are logged as alerts. This is a safety net — it should never fire in normal operation, but it catches bugs before they compound.

Incremental Rollout Plan

The data model is designed to support all three phases from day one — no breaking schema migrations needed. New capabilities are enabled by activating application-layer features on top of the existing tables.

Phase 01 — Now

Employee-Operated Core

  • Member registration & KYC
  • Savings deposits (by employee)
  • Interest credit to account
  • Loan registration & disbursement
  • EMI payment recording
  • Expense management
  • Member & portfolio reports
  • Audit trail
Phase 02 — Member Portal

Self-Service Visibility

  • Members login via Google/Apple/Phone
  • View own profile, savings, loan
  • View transaction history
  • Download statements (PDF)
  • Receive push notifications (EMI due)
  • Member-initiated deposits (eSewa/Khalti)
  • Mobile PWA home screen install
Phase 03 — Native Apps

Full Digital Lending

  • Android & iOS apps (same API)
  • Members apply for loans in-app
  • Employee loan review queue
  • Document upload from phone camera
  • Biometric login support
  • SMS/push EMI reminders
  • Advanced analytics dashboard
✅
Schema stability guarantee Phase 2 and 3 require zero breaking changes to this schema. New feature flags, notification preferences, and app-specific settings can live in the existing metadata JSONB columns or in small new satellite tables (e.g. device_tokens for push notifications) that attach via FK to users.id.
🏛️
Nepal-specific considerations Use Bikram Sambat (BS) fiscal year in the fiscal_year field (e.g. "2081-82"). Store all dates in UTC but expose in NPT (+05:45). Consider NRB compliance fields for regulated SACCOs. Nepali calendar conversions should be handled at the application layer, not the DB.
